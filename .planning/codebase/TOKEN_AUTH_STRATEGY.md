# 认证与令牌流转规范 (Token Auth Strategy)

> **受众**: 负责对接前端、移动端的开发人员，以及需要理解底层安全架构的后端骨干成员。
> **核心机制**: 无状态 JWT (Access) + 强状态单次有效追踪令牌 (Refresh Token Family)

## 1. 令牌双轨制架构设计

本项目采用 **双 Token 分离架构**，将“高性能验证”与“高频安全管控”解耦。

### 1.1 Access Token (访问令牌)
- **介质**: JWT (JSON Web Token)，采用 HS256 签名。
- **职责**: 用于一切常规业务（例如请求订单列表、修改资料等），是前端挂载在请求头 `Authorization: Bearer <Token>` 里的唯一凭证。
- **存储**: 无状态。服务器**不存储**该令牌，仅凭 CPU 计算验证其数据签名及有效性。
- **生命周期**: 短效，默认 **30 分钟**。不可由后台主动注销。

### 1.2 Refresh Token (刷新令牌)
- **介质**: Opaque String（无规则的高熵随机字串，长度 32 字节变种）。
- **职责**: 仅用于在 Access Token 过期时，换取一组全新的访问/刷新令牌。
- **存储**: 强状态。存放在服务器内存缓存（Redis）中。
- **生命周期**: 长效，默认 **7 天**。具备“单次有效（Token Rotation）”及“会话族谱监控”能力。

---

## 2. 存储结构解析 (基于 Redis 原生的防盗底座)

为了防范诸如“Token 被黑客截获盗用”、“被封禁用户仍具备续期能力”等高阶威胁，系统依靠 Redis 搭建了一套 **“会话族谱（Token Family）”** 状态树存储：

- `refresh_token:<rand_str>` -> `"Session_ID:User_ID"` (标记该凭证隶属哪个会话家族)
- `session_active:<Session_ID>` -> `<rand_str>` (家族灯塔：监控当前这个 Session 合法的、依然在役的令牌字符串是谁)
- `consumed_token:<rand_str>` -> `"Session_ID:User_ID"` (钓鱼耻辱柱：曾经合法，但已经被消耗或正在被并发攻击的标记废票)

通过这三个数据结构的指针挪动，系统实现了严丝合缝的安全隔离。

---

## 3. 标准操作流程与场景解析

### 场景一：用户注册或登录 (新建族谱)
当用户输入正确的手机号验证过关后：
1. **生成基因**：系统在后台凭空创造一个新的 `Session_ID` (由 UUID v7 生成)。
2. **生成票据**：系统计算生成 JWT (Access Token) 和一长串随机乱码 (Refresh Token)。
3. **入库记录**：将这对挂在了 `Session_ID` 下的数据推入 Redis 锁死关系。
4. **返回**：系统向前端下发这两张明文票据。前端通常应将它们储存在 LocalStorage 或内存中。

### 场景二：用户进行常规的高频操作（30分钟内）
1. 前端带着 `Access Token` 访问业务接口。
2. 系统中间件拦截器仅耗时不到 1 毫秒验算 JWT 的合法性（包含过期时间）。
3. 因不穿透数据库也不查 Redis，保障了承载百万并发的超级性能。

### 场景三：Token 过期时的安全静默刷新 (无操作感知)
当距离登录已超过 30 分钟，前端拿着 Access Token 请求会被 `401 Unauthorized` 拦截。
此时前端应**静默发起拦截器重试机制**：
1. 前端拿出压箱底的 `Refresh Token (记为票A)` 调用 `/api/v1/auth/refresh`。
2. **原子转移**：Redis 使用原生的 `RENAME` 操作，不仅保证千分之一秒并发只会通过一个，同时将 `票A` 的原始身份变更为“已消费（consumed）”状态。
3. **跨表审判**：不仅过 Redis，系统强制拉取 PostgreSQL 核心库，确认该用户是否被 `is_active=False` (封禁) 或 `is_deleted=True` (软删)。如果被除名，当场拒签！
4. **签发新票**：如果一切合法，系统生成全新的 `Access Token` 和 `Refresh Token (记为票B)`，并告诉 Redis 这个 Session 族谱现在公认合法的票是票 B。

### 场景四：黑客拦截盗用与“诛连九族”体系 (防御实演)
> 若用户在不安全网关导致其手里的 `刷新票 A` 被黑客偷偷复制并截获。

**发生阶段（黑客先手）**：
1. 黑客抢先用 `票A` 发起 `/refresh` 续签，成功拿到了最新的合法 `票B` 并在远端登录窃用。系统将 `票A` 挂上 Redis 的钓鱼耻辱柱并记录了对应的 Session。

**核爆阶段（失主上线检测）**：
1. 第二天，真正的失主点开 APP，前端不知情地用已被黑客消耗过的废票 `票A` 发起 `/refresh`。
2. **触发警报**：Redis 去查 `票A` 发现 RENAME 失败，反查后得知 `票A` 出现在名为 `consumed_token` 的耻辱柱上，证明发生了 **Token 盗用或重放攻击**。
3. **诛连销毁**：系统从耻辱柱上顺藤摸瓜揪出被盗用的 `Session_ID`，找到目前还在被黑客猖狂使用的那张合法的 `票B`。不管三七二十一，**当场从 Redis 中销毁 `票B` 及该家族根节点！**
4. **处理结果**：黑客和用户同时爆除 `401 TOKEN_THEFT_DETECTED` 强制下线，全系列通行设备变砖。用户只需修改密码并重新正常登录，而黑客的企图彻底落空。

### 场景五：用户主动退出登录
前端请求 `/api/v1/auth/logout`：
1. 系统查询 `Refresh Token` 背后的 `Session_ID`。
2. 从缓存中抹除该 `Session_ID` 以及隶属的当前合法标识和 Token 本体。此后该链路上的一切历史签发票据全部物理作废。
