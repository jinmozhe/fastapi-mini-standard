## 1. 体系全景概述

本系统建立了一套**配置驱动**的会员等级与分销网络体系。其核心理念是：

- **等级由后台运营自由配置**，不需要修改代码即可调整升级条件、折扣率、分佣比例。
- **升降级由系统自动计算**，当用户的消费、订单、邀请等指标发生变化时，系统实时重新评估等级。
- **分佣与奖励挂钩推荐关系**，每当推荐链路上的下级产生消费或升级事件，上级可按规则获得佣金或现金奖励。

```
┌─────────────────────────────────────────────────────┐
│                    运营后台 (B端)                      │
│  配置: 等级名称 / 升级条件 / 折扣 / 分佣 / 奖励        │
└────────────────────────┬────────────────────────────┘
                         │ 写入 user_levels 表
                         ▼
┌─────────────────────────────────────────────────────┐
│               规则引擎 (LevelService)                 │
│  • 升级条件 AST 匹配                                  │
│  • 分佣解析 (% 或固定金额)                             │
│  • 奖励分发 (上级 1,2,3 层)                           │
└────────────────────────┬────────────────────────────┘
                         │ 读取用户指标
                         ▼
┌─────────────────────────────────────────────────────┐
│         用户进度面板 (user_level_profiles)              │
│  total_consume / total_buy_number / total_invite     │
│  + inviter_id (推荐链路)                              │
└─────────────────────────────────────────────────────┘
```

## 2. 核心概念与术语表

| 术语 | 含义 | 示例 |
|------|------|------|
| **等级 (Level)** | 系统定义的会员身份阶梯 | 五星会员、六星会员 |
| **rank_weight** | 等级的数值权重，数字越大等级越高 | 1, 2, 3, 4 |
| **直推 (first)** | 我直接邀请注册的用户 | A 邀请了 B，B 是 A 的直推 |
| **间推 (second)** | 我的直推邀请的用户 | B 邀请了 C，C 是 A 的间推 |
| **其他层级推荐 (other)** | 我的间推邀请的用户 | C 邀请了 D，D 是 A 的其他层级推荐 |
| **upgrade_rules** | 升级条件规则 (JSONB)，支持 AND/OR 组合 | 消费满 10 万 且 邀请满 10 人 |
| **commission_rules** | 分佣配置 (JSONB)，按下级等级和层级计算 | 直推消费返 3%，间推返 2% |
| **reward_rules** | 升级奖励配置 (JSONB)，下级升级时一次性发放 | 直推升为六星会员奖励 100 元 |
| **is_manual** | 人工锁定标识，为 true 时跳过自动升降级 | 客诉补偿强制升级后锁定 |

## 3. 等级配置规则详解

### 3.1 等级定义

每个等级由后台管理员在 B 端创建，包含以下核心要素：

| 属性 | 说明 | 示例值 |
|------|------|--------|
| name | 等级名称 | 六星会员 |
| rank_weight | 排序权重 (唯一，越大越高) | 2 |
| discount_rate | 下单折扣率 | 0.95 (即 95 折) |
| upgrade_rules | 升级条件 (JSONB) | 见下文 |
| commission_rules | 分佣规则 (JSONB) | 见第 7 节 |
| reward_rules | 奖励规则 (JSONB) | 见第 8 节 |

### 3.2 升级条件配置 (upgrade_rules)

升级条件采用**规则树 (AST)** 格式，支持任意层级的嵌套 AND / OR 组合。

**运营人员可以自由组合以下维度：**

| 指标名称 (metric) | 含义 | 数据来源 |
|-------------------|------|---------|
| `total_consume` | 历史累计消费净金额 | user_level_profiles |
| `total_buy_number` | 历史累计订单数 | user_level_profiles |
| `total_invite_number` | 累计邀请注册人数 | user_level_profiles |
| `total_points` | 历史累计获得积分 | user_level_profiles |

**示例 1：单一条件 — 消费满 10 万即可升级**
```json
{
  "op": "OR",
  "conditions": [
    {"metric": "total_consume", "operator": ">=", "value": 100000}
  ]
}
```

**示例 2：多条件且 — 消费满 10 万 并且 邀请满 10 人**
```json
{
  "op": "AND",
  "conditions": [
    {"metric": "total_consume", "operator": ">=", "value": 100000},
    {"metric": "total_invite_number", "operator": ">=", "value": 10}
  ]
}
```

**示例 3：复合混搭 — (消费满 10 万 且 邀请满 10 人) 或者 (订单满 200 笔)**
```json
{
  "op": "OR",
  "conditions": [
    {
      "op": "AND",
      "conditions": [
        {"metric": "total_consume", "operator": ">=", "value": 100000},
        {"metric": "total_invite_number", "operator": ">=", "value": 10}
      ]
    },
    {"metric": "total_buy_number", "operator": ">=", "value": 200}
  ]
}
```

**支持的运算符：** `>=`、`>`、`<=`、`<`、`==`

## 4. 升级机制

### 4.1 触发时机

系统在以下业务事件发生后，自动调用 `LevelService.evaluate_user_level(user_id)` 重新计算：

| 触发事件 | 说明 |
|----------|------|
| 订单支付成功 | total_consume + 订单金额，total_buy_number + 1 |
| 新用户通过邀请注册 | 推荐人的 total_invite_number + 1 |
| 积分到账 | total_points + 获得积分 |

### 4.2 计算流程

```
1. 读取用户当前的 4 项指标 (consume / buy / invite / points)
2. 获取全部启用等级 (按 rank_weight 降序排列)
3. 从最高等级开始，逐级将用户指标喂入 upgrade_rules 规则树
4. 找到第一个返回 TRUE 的等级 → 这就是该用户应处的等级
5. 如果新等级的 rank_weight > 当前等级 → 触发 UPGRADE
6. 写入 user_level_records 变动记录
```

### 4.3 升级后效果

- 用户的 `level_id` 刷新为新等级
- 后续下单享受新等级的 `discount_rate` 折扣
- 该用户的上级推荐人收到 `reward_rules` 中配置的升级奖励（见第 8 节）

## 5. 降级机制

### 5.1 触发时机

| 触发事件 | 说明 |
|----------|------|
| 订单退款成功 | total_consume - 退款金额，total_buy_number - 1 |
| 被邀请人退出/注销 | 推荐人的 total_invite_number - 1 |

### 5.2 计算流程

与升级流程完全相同。系统重新从最高等级开始逐级匹配：
- 如果用户当前等级的规则已不再满足 → 自动跌落到能匹配的最高等级
- 写入 `user_level_records`，标记 `change_type = DOWNGRADE`

### 5.3 降级保护

- **人工锁定的用户不会被降级**：如果 `is_manual = true`，系统直接跳过该用户，保持其等级不变。
- 降级不影响历史已获得的佣金和奖励（已到账的钱不回收）。

## 6. 推荐关系与层级定义

### 6.1 推荐链路

每个用户在注册时可以绑定一个推荐人（`inviter_id`），形成链式关系：

```
用户 A (我)
  └── 用户 B (A 的直推 first)    ← A 邀请 B 注册
        └── 用户 C (A 的间推 second)    ← B 邀请 C 注册
              └── 用户 D (A 的其他级 other)    ← C 邀请 D 注册
                    └── 用户 E (超出范围，A 无关)
```

### 6.2 层级上限：最多 3 层

| 层级 | 字段名 | 关系描述 | 是否参与分佣 |
|------|--------|---------|-------------|
| 第 1 层 | `first` | 我直接邀请的人 | ✅ |
| 第 2 层 | `second` | 我的直推邀请的人 | ✅ |
| 其他层 | `other` | 我的间推邀请的人 | ⚠️ 需谨慎（见合规章节） |
| 第 4+ 层 | — | 与我无推荐收益关系 | ❌ |

### 6.3 层级追溯算法

当用户 D 发生消费时，系统向上追溯推荐链路：
1. D 的 `inviter_id` → 找到 C（C 是 D 的 1 层推荐人）
2. C 的 `inviter_id` → 找到 B（B 是 D 的 2 层推荐人）
3. B 的 `inviter_id` → 找到 A（A 是 D 的 3 层推荐人）
4. 到此截断，不再继续。

## 7. 订单消费分佣规则

### 7.1 触发场景

当用户下单支付成功后，系统为其**所有上级推荐人**（最多 3 层）计算佣金。

### 7.2 规则配置格式 (commission_rules)

存储在**推荐人所处等级**的 `commission_rules` 字段中。

```json
[
  {
    "rank": 1,
    "first": "3%",
    "second": "2%",
    "other": "0"
  },
  {
    "rank": 2,
    "first": "5%",
    "second": "3%",
    "other": "0"
  }
]
```

**字段含义：**

| 字段 | 含义 |
|------|------|
| `rank` | 消费者（下级购买者）所在等级的 `rank_weight` |
| `first` | 买家是我的直推时，我获得的佣金 |
| `second` | 买家是我的间推时，我获得的佣金 |
| `other` | 买家是我的其他层级推荐时，我获得的佣金 |

**数值解析规则：**
- 含 `%` 号 → **按订单金额百分比计算**。例如 `"3%"` = 订单金额 × 0.03
- 纯数字 → **固定金额**。例如 `"100"` = 直接发 100 元

**同一行内允许 % 和固定金额混用。**

### 7.3 完整计算示例

**场景设定：**
- 用户 A (八星会员, rank_weight=4) 邀请了 → 用户 B (六星会员, rank_weight=2)
- 用户 B 邀请了 → 用户 C (五星会员, rank_weight=1)
- 用户 C 下了一笔 1000 元订单

**A 的八星会员等级配置中 commission_rules：**
```json
[{"rank": 1, "first": "10%", "second": "5%", "other": "0"}]
```

**B 的六星会员等级配置中 commission_rules：**
```json
[{"rank": 1, "first": "3%", "second": "2%", "other": "0"}]
```

**计算过程：**

| 推荐人 | 与买家 C 的关系 | 推荐人等级的规则 | 匹配行 (C 的 rank=1) | 取哪个字段 | 佣金金额 |
|--------|---------------|-----------------|---------------------|-----------|---------|
| B | first (直推) | B 的等级规则 | rank=1 那行 | `first="3%"` | 1000 × 3% = **30 元** |
| A | second (间推) | A 的等级规则 | rank=1 那行 | `second="5%"` | 1000 × 5% = **50 元** |

> **核心逻辑总结：** 系统读取的是**推荐人自己等级**的 commission_rules，然后按**买家的 rank** 匹配行，再按**层级关系**选取 first/second/other 列。

## 8. 下级升级奖励规则

### 8.1 触发场景

当下级用户**等级发生升级 (UPGRADE)** 时，系统为其所有上级推荐人（最多 3 层）计算一次性奖励。

### 8.2 规则配置格式 (reward_rules)

存储在**推荐人所处等级**的 `reward_rules` 字段中。

```json
[
  {
    "rank": 2,
    "first": "100",
    "second": "50",
    "other": "0"
  },
  {
    "rank": 3,
    "first": "500",
    "second": "200",
    "other": "0"
  }
]
```

### 8.3 字段含义

| 字段 | 含义 |
|------|------|
| `rank` | 下级**升级后达到的等级**的 `rank_weight` |
| `first` | 该下级是我的直推时，我获得的奖励 |
| `second` | 该下级是我的间推时，我获得的奖励 |
| `other` | 该下级是我的其他层级推荐时，我获得的奖励 |

### 8.4 完整计算示例

**场景设定：**
- 用户 A (八星会员) 邀请了 → 用户 B (七星会员)
- 用户 B 邀请了 → 用户 C
- 用户 C 刚刚从五星升级为**六星会员 (rank_weight=2)**

**A 的八星等级 reward_rules：**
```json
[{"rank": 2, "first": "200", "second": "100", "other": "0"}]
```

**B 的七星等级 reward_rules：**
```json
[{"rank": 2, "first": "100", "second": "50", "other": "0"}]
```

**奖励分发：**

| 推荐人 | 与 C 的关系 | 匹配行 (C 升到 rank=2) | 取哪个字段 | 奖励金额 |
|--------|-----------|----------------------|-----------|---------|
| B | first (直推) | rank=2 那行 | `first="100"` | **100 元** |
| A | second (间推) | rank=2 那行 | `second="100"` | **100 元** |

> **注意：** 奖励是一次性发放，同一个下级同一次升级事件只触发一次。降级不发奖励。

## 9. 商品独立分佣（优先级最高）

> ⚠️ 此功能为**未来预留架构**，当前版本尚未实现。

当某个商品在后台被配置了独立的佣金金额时，该商品的订单分佣将**完全绕过**等级体系中的 `commission_rules`。

**优先级规则：**
```
商品独立佣金 > 等级 commission_rules
```

这意味着：即使推荐人是八星会员，配置了 10% 的分佣比例，但如果该商品自身设定了"每单固定返 5 元"，则最终按 5 元计算，不再读取等级规则。

## 10. 人工干预机制

后台管理员可以绕过所有自动计算规则，强制将任何用户指定为任何等级。

### 10.1 强制指定等级

- **接口：** `POST /api/v1/admin/user_levels/users/{user_id}/override`
- **效果：**
  1. 用户的 `level_id` 直接更新为指定等级
  2. `is_manual` 标志位设为 `true`
  3. 写入一条 `change_type = MANUAL` 的变动记录
  4. **此后该用户不再参与自动升降级计算**

### 10.2 解除人工锁定

- **接口：** `POST /api/v1/admin/user_levels/users/{user_id}/release`
- **效果：**
  1. `is_manual` 重置为 `false`
  2. 下一次触发 evaluate 时，系统将重新计算该用户的等级

### 10.3 使用场景

| 场景 | 操作 |
|------|------|
| 客诉补偿：用户投诉后给予 VIP 待遇 | override 到高等级 |
| 合作达人：签约 KOL 直接给最高等级 | override + 锁定 |
| 异常纠正：发现数据错误后恢复自动 | release 解锁 |

## 13. 合规与法律红线

### 13.1 适用法律

- 《禁止传销条例》（国务院令第444号）
- 《最高人民法院、最高人民检察院、公安部关于办理组织领导传销活动刑事案件适用法律若干问题的意见》

### 13.2 要求摘要

> 组织内部参与传销活动人员在**三十人以上**且层级在**其他级以上**的，应当对组织者、领导者追究刑事责任。

### 13.3 系统合规措施

| 措施 | 说明 |
|------|------|
| 推荐层级硬限制 3 层 | 代码层面只追溯 first/second/other，超出自动截断 |
| 第其它层建议不发佣金 | 配置时 `other` 建议填 `"0"` |
| 佣金来源于商品利润 | 推荐佣金从平台营销费用出，非下级用户缴纳 |
| 无入门费 | 注册会员免费，升级靠消费而非缴费 |

### 13.4 ⚠️ 绝对红线

> **微信/支付宝/聚合支付商户如果被举报存在额外级分佣行为，平台将永久封禁商户号并冻结资金。无论第其他级金额大小，哪怕 1 分钱。**

**因此强烈建议：** 在实际运营中，所有等级的 `other` 字段一律配置为 `"0"`。系统架构层面保留其他级能力，但业务运营层面严格限制在二级分销以内。

## 14. 完整业务流程图

### 14.1 用户消费 → 升级 + 分佣 流程

```
用户 C 下单支付成功 (金额: ¥1000)
    │
    ├─① 更新 C 的 user_level_profiles
    │     total_consume += 1000
    │     total_buy_number += 1
    │
    ├─② 调用 LevelService.evaluate(C)
    │     ├─ 逐级匹配 upgrade_rules (从最高等级开始)
    │     ├─ 如果匹配到更高等级 → UPGRADE
    │     │     ├─ 更新 C 的 level_id
    │     │     ├─ 写入 user_level_records
    │     │     └─ 触发 reward_rules (见 ④)
    │     └─ 如果无变化 → 结束
    │
    ├─③ 分佣计算 (向上追溯推荐链)
    │     ├─ B 是 C 的 first → 查 B 的等级 commission_rules
    │     │     └─ 匹配 rank=C的rank → 取 first 列 → 计算佣金
    │     ├─ A 是 C 的 second → 查 A 的等级 commission_rules
    │     │     └─ 匹配 rank=C的rank → 取 second 列 → 计算佣金
    │     └─ 佣金分别入账到 A, B 的钱包 (user_wallets, 未来)
    │
    └─④ 如果 C 发生了升级 (UPGRADE)
          ├─ B 是 C 的 first → 查 B 的等级 reward_rules
          │     └─ 匹配 rank=C升级后的rank → 取 first 列 → 发放奖励
          └─ A 是 C 的 second → 查 A 的等级 reward_rules
                └─ 匹配 rank=C升级后的rank → 取 second 列 → 发放奖励
```

### 14.2 用户退款 → 降级 流程

```
用户 C 申请退款成功 (金额: ¥500)
    │
    ├─① 更新 C 的 user_level_profiles
    │     total_consume -= 500
    │     total_buy_number -= 1
    │
    ├─② 调用 LevelService.evaluate(C)
    │     ├─ 逐级匹配 upgrade_rules
    │     ├─ 如果当前等级条件不再满足 → DOWNGRADE
    │     │     ├─ 跌落到能匹配的最高等级
    │     │     └─ 写入 user_level_records
    │     └─ 如果仍满足 → 等级不变
    │
    └─③ 降级不触发奖励，不回收已发佣金
```
